Trabajo Final Integrador — TallerSoft: Sistema ERP para Talleres de Servicio Técnico
Proyecto TallerSoft
Materia / Instancia Trabajo Final Integrador
Estudiante Fabrizio Leonel Ludueña
Fecha Junio 2026
El presente documento da cumplimiento a la política institucional de transparencia en el uso de herramientas de inteligencia artificial durante el desarrollo de trabajos académicos. Se declaran, con carácter exhaustivo y honesto, las herramientas de IA utilizadas a lo largo del ciclo de vida del proyecto, el propósito concreto de cada una, los fragmentos de código o diseño que tuvieron origen en dichas herramientas, las partes desarrolladas íntegramente por el estudiante, y las limitaciones o problemas encontrados con cada tecnología.
01

Herramientas de IA utilizadas

Claude
Anthropic
Asistente conversacional utilizado para la conceptualización del sistema, definición de la arquitectura de microservicios, resolución de problemas técnicos y aceleración de tareas repetitivas durante todo el desarrollo.
Stitch
Google
Herramienta de generación de interfaces utilizada para producir la base visual de las pantallas del frontend Angular (Kanban, dashboard, formularios, listados).
Gemini
Google
Modelo de lenguaje multimodal utilizado para mejorar la distribución y maquetado profesional de los campos en los formularios del frontend.
02

Para qué se utilizó cada herramienta

CLAUDE
Conceptualización, arquitectura y decisiones técnicas
Se utilizó Claude para dar forma a la idea central de TallerSoft: definir los módulos necesarios y delimitar el alcance del MVP. También se discutió la arquitectura de tres microservicios (Core Service en Java + Analytics Service en Python + API Gateway) y las razones de cada decisión: por qué Python separado para analytics, por qué JWT stateless, por qué un Gateway como punto de entrada único.
CLAUDE
Resolución de errores y tareas repetitivas de desarrollo
Se consultó para resolver errores puntuales: excepciones de Spring Security, configuración de CORS e integración con MercadoPago. Además se utilizó para acelerar tareas repetitivas (generación de DTOs, mappers, métodos CRUD estándar y configuraciones boilerplate) permitiendo focalizarse en la lógica de negocio.
CLAUDE
Asistencia en queries, tests y corrección de bugs
Asistió en la escritura de las consultas SQLAlchemy del Analytics Service (métricas de técnicos, evolución de caja, stock crítico, órdenes sin movimiento), en la generación de la estructura base de la suite de tests (JUnit 5 + Mockito, pytest, Jasmine/Karma), y apoyó la identificación de los tres bugs críticos del Sprint 6 (ExpiredJwtException sin capturar, validación de cliente inactivo, manejo de errores del asistente IA).
STITCH
Generación de la base visual del frontend Angular
Google Stitch se utilizó para generar el esqueleto inicial de las pantallas del frontend: el panel Kanban de órdenes de trabajo, el dashboard de KPIs, el módulo de clientes y el módulo de stock. Las pantallas generadas (HTML/CSS sin lógica Angular real, con datos hardcodeados) sirvieron como punto de partida visual sobre el que se trabajó la integración con el backend.
GEMINI
Distribución de campos en formularios
Gemini se utilizó para mejorar la experiencia de usuario en los formularios de alta y edición (clientes, equipos, órdenes de trabajo, repuestos). Se consultó cómo organizar los campos de forma lógica y profesional, qué campos agrupar visualmente en la misma fila, qué validaciones mostrar en línea y cómo ordenar la información para reducir la carga cognitiva del usuario final.
03

Partes del código generadas por IA

Claude · Backend Java
Estructura inicial de entidades JPA, DTOs y configuración de seguridad
Se generaron esqueletos de entidades como OrdenTrabajo, Cliente, Repuesto y Cobro, así como sus respectivos DTOs de request y response. También se generó la configuración inicial de SecurityConfig.java y la lógica del JwtAuthenticationFilter.
Stitch · Frontend Angular
Componentes visuales base de las pantallas principales
Se generaron los templates HTML y estilos base del Kanban de órdenes, el dashboard con tarjetas de KPIs, el layout general de la aplicación (sidebar, header) y las vistas de listado de clientes y stock. Estos componentes fueron el punto de partida visual antes de conectar la lógica real del backend.
Claude · Analytics Service
Estructura del microservicio Python y system prompt del asistente IA
Se generó la estructura de carpetas del servicio FastAPI, la función obtener_contexto_taller() que recopila métricas de la base de datos, y el SYSTEM_PROMPT del asistente que instruye al modelo a responder en español rioplatense usando solo datos reales del taller.
Gemini · Frontend Angular
Layout y agrupación de campos en formularios
Se generaron propuestas de layout para los formularios de alta de clientes, equipos y órdenes de trabajo: agrupación de campos en grillas, orden de presentación, jerarquía visual y organización de la sección de información del equipo vs. la falla reportada.
Claude · Boilerplate general
Código repetitivo y estructural
Se utilizó Claude para generar código estructural repetitivo: DTOs adicionales, mappers entre entidades y DTOs, métodos CRUD estándar de repositorios y servicios, y configuraciones boilerplate de Spring Boot (application.yml, beans, excepciones comunes). Este código fue revisado y adaptado antes de integrarse al proyecto.
Claude · Backend Java
Generación de presupuesto PDF con iText 7
Se generó la estructura inicial del PresupuestoPdfService: secciones del documento (encabezado del taller, datos del cliente y equipo, descripción del trabajo, tabla de repuestos con filas alternadas y total), formateo de moneda local y configuración del documento A4 con iText 7.
Claude · Backend Java
Estructura base de los tests de integración
Se generó el esqueleto de los tests de integración AuthControllerIntegrationTest y WebhookControllerIntegrationTest: configuración del contexto Spring Boot de prueba, setup de datos, y estructura de escenarios para el flujo de autenticación JWT y la validación de webhook HMAC-SHA256.
Claude · Analytics Service
Consultas SQLAlchemy extendidas y serialización de contexto
Se generaron las consultas SQLAlchemy ampliadas del servicio: rechazos_cobros, conversion_presupuesto, recurrencia_clientes, tiempo_promedio_por_estado, stock_por_categoria y get_repuestos_mas_usados. También se generó la función _construir_contexto_texto() que serializa el diccionario de contexto como texto estructurado antes de enviarlo al modelo Groq.
Claude · Analytics Service
Sistema de alertas operativas
Se generó el alertas_service.py: evalúa 5 condiciones de alerta en tiempo real (órdenes sin movimiento ≥5 días, alta prioridad parada ≥2 días, stock crítico, cobros rechazados del día, tasa de conversión <60%), asigna severidad danger/warn, y mantiene en memoria el estado leído/no leído por ID de alerta.
Claude · Frontend Angular
Dashboard con 4 pestañas, polling automático y módulo de usuarios
Se generó la estructura del DashboardComponent con 4 tabs (Operativo, Financiero, Clientes, Stock), carga paralela con forkJoin() y actualización automática de alertas cada 2 minutos usando interval() + startWith(0). También se generó la estructura base del módulo de usuarios del frontend: listado con tabla, y dialogs reutilizables de creación y edición de usuarios.
Claude · API Gateway
Filtros globales de seguridad HTTP
Se generó la configuración de default-filters en Spring Cloud Gateway: headers de seguridad X-Frame-Options: DENY, X-Content-Type-Options: nosniff y X-XSS-Protection aplicados automáticamente a todas las respuestas del sistema.
04

Partes desarrolladas íntegramente propias

Lógica de negocio completa del Core Service
Toda la lógica de negocio fue escrita y validada manualmente: las reglas de transición de estados de órdenes (PENDIENTE → EN_PROCESO → LISTO → ENTREGADO), la validación de cliente activo al crear una orden, el descuento atómico de stock al agregar repuestos y el cálculo del presupuesto acumulado.
Integración con MercadoPago
La integración completa con MercadoPago fue desarrollada manualmente: validación del webhook por firma HMAC-SHA256, idempotencia del cobro y flujo QR de punto de venta.
Integración frontend–backend y guards de seguridad
La conexión de los componentes generados por Stitch con los servicios HTTP reales, el JwtInterceptor, el AuthGuard y el RoleGuard fueron implementados manualmente. Lo mismo aplica a la lógica de drag & drop del Kanban con rollback optimista en caso de error.
Módulo de caja (frontend)
El módulo de caja del frontend fue desarrollado íntegramente: vista de caja diaria con desglose de ingresos por medio de pago, componente CobrarOrden para iniciar el flujo de cobro desde una orden lista, y consumo de los endpoints de cobros del Core Service.
Chat flotante y persistencia de historial de conversaciones
El ChatFlotanteComponent (widget accesible desde cualquier pantalla de la app) y el ChatHistoryService (gestión de sesiones de chat con persistencia en localStorage, creación, eliminación y navegación de conversaciones) fueron diseñados e implementados íntegramente. Representan una capa UX propia sobre la API del asistente IA.
Iteración del system prompt del asistente IA
Se realizaron múltiples ajustes manuales al system prompt hasta lograr que el asistente no alucinara datos y respondiera exclusivamente con el contexto provisto por obtener_contexto_taller().
Arquitectura de excepciones del Core Service
Se diseñó e implementó íntegramente la jerarquía de excepciones de dominio: InsufficientStockException, InvalidStateTransitionException, MissingDiagnosticException, EntityNotFoundException, CobrosException e InvalidCredentialsException, junto con el GlobalExceptionHandler que las mapea a respuestas HTTP estructuradas. También se creó la anotación personalizada @CurrentUser para inyectar el usuario autenticado en los controllers sin acoplarse a Spring Security directamente.
Módulo de usuarios completo (backend)
El módulo de gestión de usuarios fue desarrollado íntegramente: entidad Usuario con roles (ADMIN, TÉCNICO, RECEPCIONISTA), UsuariosController, UsuarioService, DTOs de request/response/update, mapper, y repositorio JPA. Incluye la lógica de alta, edición y desactivación de usuarios con restricciones por rol.
Modelado de la base de datos
El diseño del esquema relacional en PostgreSQL 16 fue realizado íntegramente por el estudiante: definición de entidades, relaciones, restricciones de integridad referencial e índices, tomando decisiones de normalización propias del dominio del taller.
Docker Compose y configuración de deploy
La orquestación de los cinco servicios (db, backend, analytics, gateway, frontend) con Docker Compose, la creación del usuario analytics_readonly con permisos de solo lectura en PostgreSQL, la configuración de variables de entorno de producción y el checklist de deploy fueron elaborados manualmente.
05

Problemas encontrados con el uso de IA

RESUELTO
El código generado por Stitch no estaba conectado a la lógica real
Las pantallas generadas por Stitch eran estáticamente correctas visualmente pero no tenían servicios Angular, no usaban HttpClient ni Observable, y los datos eran hardcodeados. Fue necesario refactorizar cada componente para conectarlo con los servicios reales del backend.
→ Resolución: se tomó el HTML/CSS generado como base visual y se reemplazó toda la lógica por código propio siguiendo las convenciones del proyecto.
RESUELTO
Claude sugería código de Spring Boot desactualizado (Boot 2.x)
Algunas sugerencias de Claude para la configuración de Spring Security usaban la API deprecada de Boot 2.x (encadenado de HttpSecurity con and()), que no compila en Spring Boot 3 / Spring Security 6. Fue necesario identificar el problema y reescribir la configuración con la API de lambdas de Boot 3.
→ Resolución: se verificó cada fragmento sugerido contra la documentación oficial de Spring Boot 3 antes de integrarlo.
APRENDIZAJE
El asistente IA del sistema generaba respuestas inventadas ("alucinaciones")
Al integrar el asistente IA del sistema (Groq/llama-3.3-70b), en las primeras pruebas el modelo respondía con datos inventados cuando no tenía suficiente contexto. Fue necesario diseñar un sistema de contexto dinámico que proveyera datos reales de la BD en cada consulta.
→ Resolución: la función obtener_contexto_taller() recopila métricas reales antes de cada consulta. El prompt indica explícitamente: "No inventes ni estimes información."
LIMITACIÓN
Gemini no conocía el contexto del proyecto al sugerir layouts
Las sugerencias de layout de Gemini para los formularios eran genéricas (no conocía que se trataba de un taller de servicio técnico ni los campos específicos de cada entidad). Fue necesario proveer contexto detallado en cada consulta y filtrar sugerencias que no aplicaban al dominio del negocio.
→ Resolución: se elaboraron prompts con contexto completo del dominio y se seleccionaron manualmente las sugerencias pertinentes.
